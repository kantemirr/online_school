## Рис. 8 — Контекстная диаграмма (IDEF0, уровень A-0)

Один функциональный блок — система как «чёрный ящик». Стрелки по сторонам (ICOM):

**Блок:** «Функционирование онлайн-школы программирования для детей».

- **Входы (Input, слева →):** заявка на регистрацию; данные профиля ребёнка; ответы на квиз / исходный код / файл проекта; данные оплаты; учебный контент (курсы, задания) от администратора.
- **Выходы (Output, → справа):** доступ к обучению; результат проверки (вердикт, баллы, фидбэк); квитанция об оплате; отчёт об успеваемости; уведомления (in-app + email); начисленные XP/достижения.
- **Управление (Control, сверху ↓):** ФЗ-152 (обработка ПДн детей); регламент обучения (последовательная разблокировка, порог зачёта 60%); тарифная политика; правила проверки и разграничения доступа (RBAC).
- **Механизмы (Mechanism, снизу ↑):** веб-приложение (FastAPI + React); СУБД PostgreSQL; кэш/очередь Redis; изолированная Docker-песочница; преподаватель (ручная проверка проектов).

**Внешние сущности (роли), взаимодействующие с системой:** Родитель, Ученик, Преподаватель, Администратор.

## Рис. 9 — Декомпозиция (IDEF0, уровень A0)

Блок A-0 раскрывается на 6 функций (слева-направо/по диагонали), связанных стрелками:

- **A1. Регистрация и управление аккаунтами** — регистрация родителя, подтверждение email, профили детей, вход, RBAC.
- **A2. Управление каталогом и контентом** — курсы → модули → уроки → задания (квиз/код/проект), публикация.
- **A3. Обучение и прохождение** — запись на курс, последовательная разблокировка уроков, прогресс.
- **A4. Проверка заданий** — автопроверка квиза, автопроверка кода в песочнице, ручная проверка проектов, ИИ-подсказки.
- **A5. Оплата и контроль доступа** — абонементы, имитация шлюза, активация доступа, квитанции.
- **A6. Аналитика и уведомления** — дашборды по ролям, отчёт об успеваемости, геймификация, уведомления.

**Ключевые межблочные потоки:**
- A1 → A3: учётные данные / активный ученик.
- A2 → A3, A4: структура курса и задания.
- A3 → A4: отправленная работа; A4 → A3: вердикт/баллы (закрытие урока).
- A5 → A3: статус оплаты (гейт доступа к платным курсам).
- A3, A4 → A6: события (завершение урока, проверка) → XP/достижения/уведомления.

## Рис. 10 — Диаграмма потоков данных (DFD)

**Внешние сущности:** Родитель, Ученик, Преподаватель, Администратор, Платёжный шлюз (имитация).

**Процессы (круги/прямоугольники):**
1. Аутентификация и аккаунты
2. Каталог и контент
3. Прохождение и прогресс
4. Проверка заданий (квиз/код/проект)
5. Оплата и абонементы
6. Аналитика и уведомления

**Хранилища данных (D):**
- D1 — Пользователи и профили (`users`, `*_profiles`)
- D2 — Каталог (`courses`/`modules`/`lessons`/`assignments`/`questions`/`code_tests`)
- D3 — Прохождение (`enrollments`, `lesson_progress`)
- D4 — Работы (`submissions`)
- D5 — Оплата (`subscriptions`, `payments`)
- D6 — Уведомления (`notifications`)
- D7 — Redis (токены/jti, кэш, очередь задач, лидерборды)

**Основные потоки данных:**
- Родитель → 1: учётные данные → D1; 1 → все: токен доступа (D7).
- Ученик → 2: фильтры → D2 → карточки/дерево курса.
- Ученик → 3: запись → D3; 3 → Ученик: карта прохождения.
- Ученик → 4: ответы/код/файл → D4; 4 ↔ песочница: прогон тестов; 4 → 3: вердикт → обновление прогресса (D3).
- Родитель → 5: checkout/оплата → D5; 5 → 3: активный абонемент (гейт доступа).
- 3/4/5 → 6: события → D6; 6 → Ученик/Родитель: уведомления, отчёты, рейтинг.

## Рис. 11 — Диаграмма деятельности (UML Activity) «Прохождение курса»

Готовый исходник: [activity_passage.mmd](diagrams/activity_passage.mmd). **Дорожки:** Ученик · Система · Воркер+Песочница.

Ключевые узлы и ветвления:
1. Старт → выбрать курс.
2. ⟨Курс бесплатный?⟩ — нет → ⟨Есть активный абонемент?⟩ нет → «запросить оплату» (выход); да → записаться. Бесплатный → записаться сразу.
3. Открыть разблокированный урок → изучить теорию/видео → ⟨Тип задания⟩:
   - **Квиз:** ответить → автопроверка → ⟨≥60%?⟩ нет ↺ / да → зачёт.
   - **Код:** отправить → (Система: очередь) → (Воркер: прогон тестов в песочнице) → вердикт → ⟨passed?⟩ нет → ИИ-подсказка → исправить ↺ / да → зачёт.
   - **Проект:** загрузить → ревью преподавателя → ⟨принято?⟩ нет → доработка ↺ / да → зачёт.
4. Завершить урок → начислить XP/streak/достижения → ⟨Есть следующий урок?⟩ да → разблокировать ↺ / нет → курс завершён.

Соответствие коду: `learning.enroll` / `ensure_lesson_unlocked` / `mark_lesson_completed`, `grading.submit_quiz` / `submit_code` / `submit_project` / `request_hint`, `payments.has_access`.

## Рис. 12 — Диаграмма состояний (UML Statechart) «Submission»

Готовый исходник: [state_submission.mmd](diagrams/state_submission.mmd). Состояния = enum `SubmissionStatus`.

- Начальное → **Создана** → выбор по типу:
  - **Квиз:** → **Проверена** (мгновенно).
  - **Код:** → **В очереди** (`queued`) → **Выполняется** (`running`) → **Проверена** (`checked`).
  - **Проект:** → **На проверке** (`pending_review`) → **Принята** (`reviewed`) | **На доработке** (`needs_revision`) → повторная отправка ↺.
- Внутри **Проверена** — подсостояния вердикта: `passed` / `partial` / `failed`.
- Финальные переходы: `Проверена(passed)` и `Принята` → урок закрывается.
- Деградация: при недоступности Docker работа доходит до `checked`, но с `result_json.unavailable` — урок не закрывается.
